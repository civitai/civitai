Auth Hub ↔ Main App — Actual Flows

As implemented on branch monorepo-bootstrap · traced from source · 2026-06-17 · companion to auth-hub-cutover-review-2026-06-17.md
Why this is convoluted (one root cause): the main app deploys to two registrable domains — civitai.com (shares the hub's .civitai.com parent) and civitai.red (does not). So the main app can't just let the browser talk to the hub. Instead it exposes a layer of same-origin /api/auth/* proxies that attach the browser's cookies, call the hub server-to-server, and re-set the cookie locally — because the hub's Set-Cookie is scoped to .civitai.com and can't cross back to .red. Every flow below routes through that proxy layer. On civitai.com some of these hops are technically redundant (the cookie is already visible) but the same code path runs on both domains.

Browser

The user's tab(s). Holds cookies, runs SessionProvider, useDomainSync, popup coordination.

Spoke = main app

Next.js on civitai.com or civitai.red. Its /api/auth/* routes are same-origin proxies + the SSR resolver.

Hub

auth.civitai.com (SvelteKit). Sole producer of session data; mints all tokens; owns login UI, swap, switch, impersonate, identity.

Shared Redis

session:data2:<userId> user cache + TOKEN_STATE revocation markers. Read by every spoke, written by the hub.

Postgres

Touched only by the hub when producing a fresh SessionUser on a cache miss.

Cookies (4)

civ-token (session), civ-device (account roster), legacy civitai-token (NextAuth fallback, read-only), orchestrator cookie.

A · Reading a session

UC1 — Server-side resolution (every SSR page & API route)

getServerAuthSession() is a 4-level precedence ladder. First match wins; it memoizes onto req.context.session. Source: src/server/auth/get-server-auth-session.ts.

flowchart TD
    REQ["Incoming request"] --> MEMO{"req.context.session already set?"}
    MEMO -->|yes| RET["return cached session"]
    MEMO -->|no| BEARER{"Authorization Bearer or token query param present?"}
    BEARER -->|yes| BT["getSessionFromBearerToken: hash to ApiKey lookup, user plus tokenScope/buzzLimit/subject"]
    BT --> RET
    BEARER -->|no| HUB{"civ-token cookie? getHubSession verifies ES256 locally"}
    HUB -->|valid| CACHE["shared Redis session:data2 by id, then hub /api/auth/identity on miss"]
    CACHE --> ROLL["maybeRollHubCookie: rolling refresh on activity"]
    ROLL --> RET
    HUB -->|none or invalid| LEG{"legacy civitai-token? getLegacySession jose JWE decode"}
    LEG -->|valid| FRESH["getSessionUser by id, fresh from cache/DB"]
    FRESH --> RET
    LEG -->|none| NULL["return null, signed out"]

    class BT,HUB,CACHE,ROLL hub
    class LEG,FRESH legacy
    classDef hub fill:#0d2741,stroke:#58a6ff,color:#e6e8ee
    classDef legacy fill:#2b2207,stroke:#d29922,color:#e6e8ee
Known issues (review): the ?token= query-string bearer is still accepted (foot-gun, logs/Referer leakage); the legacy JWE path enforces neither issuer nor audience (B3) and has no revocation gate, so during the cutover a legacy-cookie holder isn't rejected by isRevoked.

UC4 — Login on civitai.red (cross-site — the convoluted one)

Same entry point detects cross-site (registrableDomain differs) and routes the landing through /api/auth/sync. The hub cookie is not readable on .red, so a swap-token round-trip mints a .red-local cookie. Source: auth-helpers.ts, spoke sync.ts, hub sync/+server.ts, hub exchange.

sequenceDiagram
    autonumber
    participant P as Popup on civitai.red
    participant H as Hub
    participant RS as red /api/auth/sync
    participant HX as Hub /api/auth/exchange
    participant PL as red /api/auth/post-login
    P->>H: hub /login, returnUrl points at red /api/auth/sync
    H->>H: authenticate, Set-Cookie .civitai.com, NOT visible to .red
    H-->>P: 302 to red /api/auth/sync
    P->>RS: GET /api/auth/sync, no swap yet
    RS-->>P: 302 to hub /api/auth/sync, callback is red /api/auth/sync
    P->>H: GET hub /api/auth/sync, top-level nav so Lax .civitai.com cookie rides along
    H->>H: locals.user set, mintSwapToken single-use 60s, callback in AUTH_SPOKE_ORIGINS
    H-->>P: 302 to red /api/auth/sync with swap token
    P->>RS: GET /api/auth/sync with swap, Referrer-Policy no-referrer
    RS->>HX: POST /api/auth/exchange with swap, server to server
    HX-->>RS: returns full civ-token
    RS->>RS: setSessionCookie, red civ-token
    RS-->>P: 302 to /api/auth/post-login
    P->>PL: GET /api/auth/post-login, civ-token now readable
    PL-->>P: side-effects, 302 to popup-done, channel DONE to origin tab
Known issues (review): the swap token travels in the URL (logs/Referer; mitigated by single-use + 60s + no-referrer), is not bound to the redeeming spoke (B4), /exchange is unauthenticated + unrate-limited, and single-use silently disables if REDIS_SYS_URL is unset (swap.ts returns success). The hub's post-login redirect allowlist uses origin.includes('civitai') (B1, open redirect).

UC4-simplified — remove the redundant re-bounce (planned near-term, no architecture change)

Today's UC4 makes the browser bounce spoke → hub → spoke → hub just to obtain the swap token: after the hub login sets its cookie, it redirects to red/api/auth/sync, which immediately redirects back to hub/api/auth/sync only to mint the swap. The hub already knows the user is authenticated the moment login completes, so it can mint the swap inline at login-completion when the returnUrl targets an allowlisted spoke /api/auth/sync — redirecting straight to red/api/auth/sync?swap=…. That removes ~2 redirects. See auth-login-simplification.md (#2).

Deferred to the OAuth migration (not now): fully replacing the swap bridge with the standard OIDC authorization-code flow, and unifying the same-site (UC3) and cross-site (UC4) paths into one. That migration is cookie-safe — setSessionCookie writes the same civ-token regardless of how it was obtained, so existing sessions survive. Steps live in docs/plans/oauth-provider-implementation-checklist.md §I; rationale in docs/auth-login-simplification.md. Until then the swap bridge stays, with its B1/B4/fail-closed blockers fixed for production.

UC5 — Cross-domain sync on any page load: useDomainSync() fires when a page loads carrying the SYNC_PARAM marker (set by a "sign in on this domain" link), stripping it and navigating to /api/auth/sync?returnUrl=… — i.e. it enters UC4 from the RS step. Source: src/hooks/useDomainSync.tsx.

UC6 — Logout

Source: handleSignOut() → src/pages/api/auth/logout.ts.

sequenceDiagram
    autonumber
    participant B as Browser
    participant L as Spoke /api/auth/logout
    participant H as Hub revoke
    B->>L: GET /api/auth/logout, callbackUrl is here
    L->>H: sessionTokenClient.revoke civ-token, best-effort, never blocks
    H->>H: mark jti invalid in TOKEN_STATE
    L->>L: clear civ-token both prefixes across host-only, AUTH_COOKIE_DOMAIN, dot-host
    L->>L: clear legacy civitai-token across domains, plus orchestrator cookie
    L-->>B: 302 to callbackUrl, reloads signed-out
Known issues (review M2): logout does not clear the civ-device cookie, so the seamless account-switch set survives logout on shared machines; and logoutAll is a no-op alias for logout (no "sign out everywhere"). Also clears only the current jti, not all of a user's sessions.
