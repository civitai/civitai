First-Party SSO — Current vs Proposed

Wire diagrams · companion to first-party-sso-vs-oauth-analysis.md & auth-hub-cutover-review-2026-06-17.md · 2026-06-17
The proposal is narrow, not a rewrite. The shared-domain cookie for the *.civitai.com family and the per-request session lookup are identical in both designs. The only place they diverge is how a cross-root domain (civitai.red) gets bootstrapped: today's hand-rolled swap-token bridge vs. a standard OIDC authorization-code bridge. Read the diagrams in that light.

Diagram 0 — Per-request session lookup IDENTICAL in both

How any spoke turns the thin cookie into a full user. The cookie carries only { sub, jti, signedAt } — the rich SessionUser is looked up by userId in shared Redis, which every domain reads (so session data is centralized even though the cookie can't cross eTLD+1).

Diagram 1 — CURRENT solution hybrid: 2 bootstrap mechanisms

Hub mints the thin token and sets a Domain=.civitai.com cookie that every subdomain inherits for free. A separate, hand-rolled swap-token bridge handles the cross-root civitai.red.

Diagram 2 — PROPOSED solution same cookie · standard bridge

The *.civitai.com path is unchanged. Only the cross-root bridge swaps the bespoke flow for a standard OIDC authorization-code + PKCE handshake — and it still ends in the same thin cookie. civitai.red is just a registered trusted client (consent skipped).

What actually changes

AspectCurrentProposed
*.civitai.com family cookie Hub sets .civitai.com; subdomains inherit free identical — kept
Per-request session lookup Thin token → shared Redis → hub identity on miss identical — kept
New subdomain app effort Read cookie + verify via SDK identical
Cross-root (.red) bootstrap bespoke swap-token mint → ?swap= redirect → /exchange standard OIDC auth-code + PKCE
Who sets the .red cookie .red server, after swap exchange .red server, after /token exchange (same actor)
Cross-root security properties hand-rolled open-redirect (B1), binding (B4), replay inherited from OIDC state / nonce / PKCE / redirect_uri
New cross-root brand effort Wire a bespoke swap bridge Register a redirect_uri (config, not code)
Code to maintain swap + exchange + sync endpoints Folded into the OAuth provider you're already building
Bottom line: keep the cheap shared-domain cookie and the thin-token lookup; replace only the hand-rolled .red bridge with the standard authorization-code flow. The swap-token flow stops being "the SSO mechanism" and is correctly understood as "the cross-root adapter" — which OIDC already does, hardened, for free.
