Civitai Monorepo Migration

Extracting our infrastructure (Postgres, Redis, ClickHouse, Axiom, telemetry) into app-agnostic packages any future app can reuse — and the domain contracts two apps must agree on — without copying connection code, re-validating env, or drifting on shared data.

pnpm workspaces 6 packages + domain layer Factory + injected behavior Per-package env schemas Moderator app: first consumer

01Overview

Why we're doing this, in one paragraph.

Today the main Next.js app owns all of its infrastructure clients directly. As we add more apps (a moderator app, an advertising service, internal tools), each one needs the same Postgres / Redis / ClickHouse connections — and we don't want to copy-paste connection setup, drift on pool tuning, or validate the same env vars three different ways.

The migration moves each piece of infrastructure into its own package under packages/. A package owns how to connect (validated env + a factory that builds the client); each app decides whether and with what behavior to instantiate it (its own logger, feature flags, etc.). The main app keeps working unchanged through thin shim files at the original import paths.

Net result: a new app gets production-grade DB/Redis/ClickHouse access in a few lines, with the exact same connection behavior and env contract as the main app.

02The layering model

Four layers, each only depending downward.

Apps  ·  main app (repo root), apps/*
Business logic. Imports infrastructure through shims / factories. Owns its own env values, loggers, and policies.
▲ imports
App shims  ·  src/server/{db,redis,clickhouse,logging,prom}/…
Thin app-owned files at the original import paths. Call the factory, inject app behavior, own HMR globals + the Next build guard, re-export the same names so call sites never change.
▲ calls factory
Base packages  ·  @civitai/{db,redis,clickhouse,axiom,telemetry}
Infrastructure only. Each owns a zod env schema + a createX() factory. Imports external npm deps only — never app code, never a sibling base package.
▲ generated types only
Contract layer  ·  @civitai/db-schema  ·  @civitai/domain
db-schema — pure generated artifact: Prisma schema, migrations, generated client / enums / models (+ Kysely types). No runtime; a Kysely app uses it without the Prisma runtime. domain — pure hand-authored vocabulary both apps must agree on: non-Prisma enums, the nsfwLevel bit-decoders, base-model identity. Both are leaves.
The only cross-package edges point downward onto the contract leaves: @civitai/db → @civitai/db-schema (imports the generated Prisma client) and @civitai/domain → @civitai/db-schema (references generated enums). Like depending on @prisma/client — not sibling coupling.

04Architecture rules

The handful of rules that keep packages reusable. Follow these when adding a package.

① Packages import external deps only

No ~/… app imports. No sibling base package. The moment a file needs app code, it's a consumer of infra, not infra — it stays in the app (or a higher-level package).

② Env values → the package's schema

Each package owns a env.ts zod schema (mirrors the app's server-schema.ts) and reads process.env through it. Validated on deploy. Connection config never crosses the boundary as plain arguments.

③ App behavior → injected as functions

Loggers, the Flipt failover resolver, the slow-query → Axiom sink: these are functions/policy the app owns, passed into the factory. Not env, not baked into the package.

④ isProd in package, isBuild in shim

NODE_ENV is universal, so packages may read it. Detecting next build is a Next-specific concern, so the build guard (env.IS_BUILD ? skip : createX()) lives in the app shim.

Mnemonic: a package answers “how do I connect?” (env + factory). An app answers “should I, and with what behavior?” (build guard + injected logger/policy + globals).

05Anatomy of a package

Every package is the same three pieces. Here's @civitai/redis end to end.

1 · env.ts — package-owned, validated config

// packages/civitai-redis/src/env.ts
import * as z from 'zod';

const schema = z.object({
  REDIS_URL: z.url(),
  REDIS_SYS_URL: z.url(),
  REDIS_TIMEOUT: z.preprocess((x) => (x ? parseInt(String(x)) : 5000), z.number().optional()),
  REDIS_CLUSTER: z.preprocess((x) => x === 'true', z.boolean().default(false)),
  /* … cluster nodes, refresh interval, flipt context … */
});

const parsed = schema.safeParse(process.env);
if (!parsed.success)
  throw new Error('[@civitai/redis] Invalid env:\n' + z.prettifyError(parsed.error));

// Normalized defaults the factory can override per call.
export const redisEnv = { url: parsed.data.REDIS_URL, /* … */ };
export type RedisConfig = typeof redisEnv;

2 · client.ts — the factory (reads env, injects behavior)

// packages/civitai-redis/src/client.ts
export function createRedisClients(
  options: Partial<RedisConfig> & {
    log?: RedisLogFn;                                 // injected app logger
    isEnhancedFailoverEnabled?: RedisFailoverResolver;  // injected Flipt policy
  } = {}
): { redis: CustomRedisClientCache; sysRedis: CustomRedisClientSys } {
  const config = { ...redisEnv, ...envOverrides };   // env defaults + per-call overrides
  // build cache + sys clients from config; failover uses the injected resolver
  return { redis, sysRedis };
}

export const REDIS_KEYS = { /* … key definitions stay in the package … */ };

3 · The app shim — wires reality, owns globals

// src/server/redis/client.ts  (main app — original import path preserved)
import { createRedisClients } from '@civitai/redis/client';
export * from '@civitai/redis/client';   // re-export keys, types, factory

const make = () => createRedisClients({
  log: createLogger('redis', 'green'),                  // THIS app's logger
  isEnhancedFailoverEnabled: (ctx) => isFlipt(FLAG, 'redis-cluster', ctx),
});

// build guard (Next-specific) + HMR singleton both live here, not in the package
const clients = env.IS_BUILD ? EMPTY : isProd ? make() : (global.__redis ??= make());
export const { redis, sysRedis } = clients;
Because the shim re-exports redis, sysRedis, and REDIS_KEYS under the same names, every existing import { redis } from '~/server/redis/client' call site keeps working unchanged.

06Using base packages in a new app

This is the payoff. A new app in apps/ gets the same infrastructure in a few lines.

1

Scaffold

Create apps/moderator/ with a package.json. pnpm-workspace.yaml already globs apps/*, so it's picked up automatically.

2

Configure env

Set the vars each package validates (DATABASE_URL, REDIS_URL, …). Missing → fail fast on boot.

3

Call factories

Instantiate clients with createX({ log }), injecting this app's logger & policies.

4

Use & ship

Same typed API as the main app — no connection code, pooling, or env validation to rewrite.

Step 1 · Set the env vars (same names the packages validate)

Each package validates its own slice of process.env on boot. Provide the same variable names the main app uses — the package's schema enforces them.

# apps/moderator/.env
DATABASE_URL=postgres://…
DATABASE_REPLICA_URL=postgres://…
REDIS_URL=redis://…
REDIS_SYS_URL=redis://…
# missing/invalid → the app fails fast on boot with a clear message

Step 2 · Build the clients via the factories

// apps/moderator/src/db.ts
import { createPrismaClients } from '@civitai/db';
export const { dbRead, dbWrite } = createPrismaClients();   // env supplies the config

// apps/moderator/src/redis.ts
import { createRedisClients } from '@civitai/redis';
export const { redis, sysRedis } = createRedisClients({
  log: myAppLogger,        // inject YOUR logger
  // no Flipt in this app? omit it — enhanced failover stays off by default
});

// apps/moderator/src/clickhouse.ts
import { createClickhouseClient } from '@civitai/clickhouse';
export const clickhouse = createClickhouseClient({ log: myAppLogger });

Step 3 · Use them — identical typed API to the main app

import { dbRead } from './db';
import { redis, REDIS_KEYS } from './redis';

const user = await dbRead.user.findUnique({ where: { id } });
await redis.packed.set(REDIS_KEYS.SOMETHING, value);
What you did not do: write pool tuning, re-implement the typed Redis wrapper, set up the TIMESTAMP parser, or re-declare env validation. The package owns all of that — so every app connects the same way.

Per-call overrides

Env supplies defaults; the factory accepts a Partial<Config> to override any of them — handy for tests, multi-instance setups, or an alternate config source.

// override an env default for this instance only
const logger = createAxiomLogger({ datastream: 'moderator-logs' });

// point a test at a throwaway DB without touching process.env
const { dbWrite } = createPrismaClients({ databaseUrl: TEST_DB_URL, replicaUrl: TEST_DB_URL });

Prefer Kysely over Prisma? Use the contract layer directly

This is why @civitai/db-schema is split out from @civitai/db. The schema is the source of truth; the Prisma runtime is just one way to query it. An app that wants Kysely (as our advertising service does) consumes the generated types and brings its own query builder — without the Prisma-client runtime.

// 1. add a prisma-kysely generator to schema.full.prisma (one block):
//    generator kysely { provider = "prisma-kysely"  output = "../src/kysely" }

// 2. apps/ads/src/db.ts — Kysely over a raw pg pool, typed by the same schema
import { Kysely, PostgresDialect } from 'kysely';
import type { DB } from '@civitai/db-schema/kysely';
export const db = new Kysely<DB>({ dialect: new PostgresDialect({ pool }) });
Status: the prisma-kysely generator isn't wired into the schema yet — this is the supported path for the first Kysely-based app to turn on. Migrations & enums/models already generate from the same schema today.

07The domain layer — sharing more than infrastructure

Infra was the easy part. The first new app forces a second, subtler kind of sharing.

Standing up the moderator app — the first app to run its own routers against the shared database — surfaced a new class of shared code. Beyond connections, two apps that write the same tables must agree on the hand-authored meaning of those columns: which bit of nsfwLevel is R-rated, what BlocklistType.LinkDomain equals, which ReportEntity keys the Report table. Keep two copies and they drift — and drift on a shared column is silent data corruption.

@civitai/domain — a new pure, hand-authored contract package, peer to @civitai/db-schema. It holds the vocabulary both apps must agree on: non-Prisma enums (BlocklistType, NotificationCategory, NsfwLevel, ReportEntity, unpublishReasons), the nsfwLevel bit-decoders (browsing levels + Flags), file-ext ↔ MediaType maps, and base-model / ecosystem identity. Its only edge is the downward dep @civitai/domain → @civitai/db-schema.

The forcing function

The rule for what becomes a package is deliberately narrow — not “is it pure” or “is it reused,” but:

Extract to a package only when duplicating the code would corrupt shared data — a hand-authored contract on a column both apps read or write. Everything else is copied into the app (generic utils, display strings) or left in the main app behind its API (feed / marketplace machinery). src/shared/ is a client/server boundary within the main app — orthogonal to cross-app packaging.

Two ways code enters the package

① Whole-file move

The file is the shared unit. Pure git mv (R100 rename, history preserved), then a re-export shim at the old path so every existing call site keeps working. Used for enums, browsingLevel.constants, flags, mime-types, basemodel.constants.

② Member extraction

A small shared member is buried in an app-coupled file (e.g. CacheTTL inside a 1,700-line env-coupled constants.ts). Carve out just that member, leave a re-export behind — the coupled file never moves. A content edit, not a rename.

08Package · copy · proxy

How every dependency of a new app gets sorted — demonstrated by the moderator app.

Tracing the moderator pages through their tRPC routers and services showed the server is ~80% cleanly separable (read queues, simple writes, Redis config, orchestrator calls) and ~20% entangled in feed / marketplace machinery. Crucially, that clean 80% is moderator-only — one consumer — so it lives in the app, not a package. Every remaining dependency sorts into one of three buckets:

📦 Package  ·  shared contract — duplication corrupts data
@civitai/domain (the vocabulary above) · persisted JSON-column shapes (ModelMeta → Model.meta, UserMeta → User.meta) · the 13-line @civitai/orchestrator client. Both apps import one source of truth.
📋 Copy  ·  generic — drift is harmless
Prisma selectors, tRPC input schemas, pagination-helpers, errorHandling, display strings. Each app keeps its own — a mismatch is cosmetic, never corrupting.
🔌 Proxy  ·  entangled — stays in the main app
Feed cache-busting, search-index sync, the image-rating game, rewards, the buzz ledger, upsertModel. The moderator app calls the main app’s tRPC instead of owning these.
Extract-clean-slice. The moderator’s functions are often buried inside giant entangled services — e.g. CSAM needs a ~20-line ClickHouse insert from the 7,982-line image.service. The fix is to lift the clean slice out into a small module; because it has one consumer, it travels with the moderator app, not into a package.
Net result: the entire moderator app adds just two packages on top of the six already shipped — @civitai/domain (shared vocabulary) and @civitai/orchestrator (the generation client). Everything else is the app’s own code, a copy, or a call back to the main app.
